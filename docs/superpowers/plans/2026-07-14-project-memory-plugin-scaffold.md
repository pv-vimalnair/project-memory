# Project Memory Codex Plugin Scaffold Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create the validated repo-local Codex marketplace, `project-memory` Plugin shell, implicitly invokable skill, and package-root boundary required before the engine implementation begins.

**Architecture:** The repository root is a one-Plugin marketplace. `plugins/project-memory/` is the only npm package and eventual installable Plugin archive. The Plugin skill is the agent-facing doorway; it invokes a deterministic engine that later plans provide. Scaffold validation must pass before foundation code is added.

**Tech Stack:** Codex Plugin manifest, Codex marketplace manifest, Codex skill metadata, Python 3.11 scaffold/validation scripts, Markdown, JSON, YAML, Git, and PowerShell.

## Global Constraints

- Approved base design: `docs/superpowers/specs/2026-07-14-project-memory-system-design.md`.
- Approved Plugin addendum: `docs/superpowers/specs/2026-07-14-project-memory-agent-plugin-design.md`.
- Repository root: `<repository-root>` or an isolated worktree of that repository.
- Plugin package root: `plugins/project-memory/`.
- Run the Plugin Creator and Skill Creator scripts; do not hand-invent their required directory contracts.
- The Plugin name, folder name, skill name, marketplace entry, package name suffix, and runtime command all use `project-memory`.
- V1 has no hooks, apps, MCP server, icon, logo, screenshots, hosted service, telemetry, or runtime network dependency.
- Omit optional manifest fields until their real files or approved public URLs exist.
- Use `UNLICENSED` during local development. A public license requires separate Pitaji approval.
- The repo-local marketplace entry is `AVAILABLE`, uses `ON_INSTALL`, has category `Productivity`, and points to `./plugins/project-memory`.
- Do not write `~/.agents/plugins/marketplace.json`, install the Plugin into Codex, publish, push, or touch a live product repository in this plan.
- The existing uncommitted root `package.json` and `tests/smoke/package.test.ts` were generated by the aborted CLI-first start. They may be safely relocated only after their paths and untracked status are verified exactly.

# Project Memory

Use the bundled engine; do not ask the user to choose folders, profiles, patterns, or record destinations.

## Start every project task

1. Locate the repository root and derive the Plugin root by moving two directories up from this skill directory.
2. Run `node <plugin-root>/scripts/project-memory.mjs agent start --root <repository-root> --json`, adding `--brief <path>` when a brief exists.
3. Treat the returned directive as authoritative for startup behavior.

For `bootstrap_review_required`:

1. Present the engine-selected root, blueprint, components, domains, adapters, source proposal, assumptions, and risks concisely.
2. Ask at most one grouped clarification only when the directive requires it.
3. Ask Pitaji for one confirmation of the complete proposal; never make him select a profile.
4. After confirmation, run the returned `init apply` arguments exactly.
5. Re-run `agent start` and continue only after it returns `resume`.

For `resume`:

1. Read every path in `reading_order` in order.
2. Read only the assigned task packet and directly referenced records after the startup set.
3. Follow the locked root profile. Never reclassify an existing product because of a task prompt.

For `blocked`, report the exact issues and perform no mutation.

## Perform work

1. Convert the user's normal outcome into engine-planned initiatives, workstreams, and task packets.
2. Obtain a claim before editing and stay inside allowed paths.
3. Treat generated views, locks, canonical events, records owned by another task, and append-only history as no-touch paths.
4. Workers may propose direction but never accept it.
5. Run every required gate exactly and retain evidence, including checks not run and why.

## Finish work

1. Produce the required completion packet with changed paths, commands, results, evidence, risks, omissions, and base revisions.
2. Validate the packet and claim through the engine.
3. If acting as a worker, return the packet to the integrator and do not update canonical truth.
4. If explicitly acting as the integrator, use the engine's validate/finalize flow. Never hand-edit generated views or append-only history.
5. Report the resulting receipt or exact failure. A failed finalization leaves canonical state unchanged.

## Safety

- Use plan operations before apply operations.
- Never replay a saved mutation without recomputing and matching its hash and current Git head.
- Never infer approval from silence, tests, or a worker completion statement.
- Never expose secrets, credentials, private transcripts, or absolute local paths in generated public artifacts.
- Never publish, deploy, message externally, or change a live product without the required explicit authority.

Read `references/agent-protocol.md` only when handling role boundaries, missing Plugin behavior, legacy import, migration, or multi-repository finalization.
